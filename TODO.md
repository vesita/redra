## 🚨 重要重构：通信协议模式更新

### 当前问题

**问题描述**: 
- 当前系统使用 "Trailer + Pack" 模式进行通信，其中Pack是具体的命令数据
- 所有数据通过Protobuf Command消息进行传输，使用Trailer协议定义数据包边界
- 需要将所有通信重构为统一的 "Trailer + Command" 模式，所有数据都包装在Command子包中

**新模式定义**: **TCMP (Trailer-Command Messaging Protocol)** - Trailer-Command消息传递协议
- **Trailer**: 协议预告信息，定义数据包边界
- **Command**: 统一的消息容器，所有数据都作为Command的子包存在
- **子包**: 具体的数据类型，如Designation、Transform、PointCloud等
- **宏指令**: 用于控制层面的消息，如数据集切换、连接确认等

**目标**: 实现TCMP (Trailer-Command Messaging Protocol) 统一通信协议

**核心概念**:
```rust
// Command消息结构
message Command {
  oneof cmd_pack {
    target.ConceptionCMD conception = 1;
    designation.DesignCMD designation = 2;
    transform.TransCMD transform = 3;
    resource.RsrcPack resource = 4;
    shape.ShapePack shape = 5;
    pointcloud.PointCloudPack point_cloud = 6;
    // ... 更多命令类型
  }
  int64 timestamp = 4;
  string command_id = 5;
}

// Trailer协议预告信息
message Trailer {
  uint32 me = 1;    // trailer自身的大小
  uint32 next = 2;  // 后续payload的大小
}
```

**设计要点**:
- **统一入口**: 所有网络通信都通过Command消息进行
- **数据封装**: 所有数据类型都作为Command的oneof子包存在
- **宏指令支持**: Command可以包含控制类命令，如数据集切换、心跳等
- **自动处理**: 在redra_proto中实现自动编码/解码、Trailer处理等功能

#### 1. 重构网络通信层

**目标**: 将所有网络发送和接收重构为TCMP模式

**需要修改的组件**:
- [ ] `redra_client` crate中的Sender/Receiver
- [ ] `redra_proto` crate中的编码/解码函数
- [ ] `net`模块中的forwarder和linker组件

#### 2. 增强Command结构

**目标**: 为Command添加宏指令支持

**需要添加的内容**:
- [ ] ControlCmd: 用于连接管理、心跳、确认等
- [ ] MetaCmd: 用于元数据传输、配置设置等
- [ ] BatchCmd: 用于批量命令处理

#### 3. 实现自动处理功能

**目标**: 在redra_proto中实现完整的TCMP处理逻辑

**需要实现的内容**:
- [ ] 自动Trailer解析
- [ ] Command编码/解码
- [ ] 错误处理和重试机制
- [ ] 数据完整性校验

#### 4. 实现计划

**阶段 1: 核心协议重构** (优先级：高)
- [ ] 修改`redra_proto`中的编码/解码逻辑

## 🚨 紧急重构：数据管理系统设计

### 当前问题

**问题描述**: 
- 当前的 `recv_and_spawn` 系统与 `record_data_frames` 系统竞争消费同一个 channel receiver
- 导致大量数据包丢失（990 个数据包中只有少数被记录）
- 无法实现完整的帧级数据管理

**临时解决方案**:
- ✅ 已禁用 `recv_and_spawn` 的实时生成功能
- ✅ 所有数据现在通过 `record_data_frames` 系统记录到 `DataRecorder`
- ✅ UI 回放控制功能已实现并可用

**需要重新设计的内容**:

#### 1. 基于帧的数据管理架构

**目标**: 实现按帧（Frame-based）的点云数据管理系统

**核心概念**:
```rust
struct DataFrame {
    frame_id: u32,           // 帧 ID
    timestamp: u64,          // 时间戳
    points: Vec<RDPack>,     // 该帧包含的所有点云数据
    metadata: FrameMetadata, // 元数据
}
```

**设计要点**:
- **实时累积**: 接收到的数据先累积到当前帧缓冲区
- **帧完成判断**: 
  - 达到预期点数
  - 或超过时间阈值
  - 或收到显式的帧结束信号
- **帧持久化**: 完成的帧保存到录制器，支持后续回放
- **按需生成**: 回放时根据当前帧动态生成/销毁场景实体

#### 2. 移除实时生成模式

**原因**:
- 实时生成导致数据被竞争消费
- 无法保证数据的完整性和可追溯性
- 不支持回放、暂停、快进等高级功能

**新流程**:
```
数据接收 → 帧累积 → 帧完成 → 保存到录制器 → UI 控制回放 → 场景生成
```

#### 3. 实现计划

**阶段 1: 核心数据结构** (优先级：高)
- [ ] 完善 `DataFrame` 结构
- [ ] 实现 `FrameManager` 资源
- [ ] 修改 `record_data_frames` 系统以帧为单位记录

**阶段 2: 帧缓冲区管理** (优先级：中)
- [ ] 实现循环帧缓冲区
- [ ] 内存优化（大数据集的分页加载）
- [ ] 帧的序列化/反序列化

**阶段 3: 回放系统集成** (优先级：中)
- [ ] `PlaybackManager` 按帧回放而非按点回放
- [ ] 场景管理器（按需生成/销毁实体）
- [ ] 平滑过渡动画

**阶段 4: UI 增强** (优先级：低)
- [ ] 帧列表/时间轴显示
- [ ] 帧详情面板
- [ ] 批量操作（导出、删除等）

## 数据回放系统 (Data Replay System)

### 目标概述

实现通过 UI 界面控制的数据回放功能，支持接收到的数据以数据帧（Data Frames）和数据帧系统（Data Frame System）的形式在 Redra 中进行回放。

### 核心需求

#### 1. 数据帧系统架构

- **数据帧定义**: 将接收到的每个数据包封装为带有时间戳和序列号的数据帧
- **帧缓冲区**: 实现循环缓冲区或队列来存储历史数据帧
- **时间同步**: 支持原始时间戳记录和回放时的时间控制

#### 2. UI 控制面板

- **播放控制**:
  - 播放/暂停/停止按钮
  - 快进/快退功能
  - 单帧前进/后退
- **速度控制**:
  - 回放速度调节（0.5x, 1x, 2x, 4x 等）
  - 实时速度滑块
- **时间轴显示**:
  - 可视化时间轴展示所有数据帧
  - 当前播放位置指示器
  - 关键帧标记

#### 3. 数据帧管理

- **帧存储策略**:
  - 内存管理与缓存策略
  - 支持大数据量的分页加载
  - 可选的持久化存储（保存到文件）
- **帧查询与过滤**:
  - 按时间范围查询
  - 按数据类型过滤
  - 搜索特定帧

#### 4. 回放模式

- **实时回放**: 按照原始时间间隔回放
- **逐帧回放**: 手动控制每一帧的显示
- **循环回放**: 支持循环播放选定的数据段
- **断点续播**: 记录播放位置，支持中断后继续

#### 5. 数据可视化增强

- **帧信息面板**: 显示当前帧的详细信息
  - 时间戳
  - 数据类型
  - 数据大小
  - 来源标识
- **统计信息**:
  - 总帧数
  - 已播放帧数
  - 剩余时间估算
  - FPS 显示

### 技术实现要点

#### Bevy ECS 集成

- 使用 Bevy 的 Resource 和 Component 系统管理回放状态
- 通过 Message 系统实现 UI 与回放系统的通信
- 利用 Bevy 的场景系统管理回放过程中的实体生成/销毁

#### 异步处理

- 数据帧的加载和解析应在后台线程进行
- 避免阻塞主渲染循环，保持流畅的 UI 响应

#### 性能优化

- 实现帧缓冲池减少内存分配
- 支持增量更新，避免全量刷新
- 优化大量数据帧的渲染性能

### 预期功能扩展

- **录制功能**: 支持将实时接收的数据保存为可回放的格式
- **导出功能**: 支持将数据帧导出为标准格式（JSON、CSV 等）
- **多通道回放**: 支持多个数据流的同步回放
- **标注系统**: 允许在回放过程中添加注释和标记

### 验收标准

- [x] UI 界面提供直观的播放控制
- [ ] 支持至少 10,000 帧的流畅回放
- [ ] 回放速度可在 0.25x 到 4x 范围内无级调节
- [ ] 时间轴响应迅速，支持快速跳转
- [ ] 内存使用合理，不会因大量数据导致崩溃
- [ ] 支持暂停、恢复而不丢失状态
- [ ] 提供完整的错误处理和边界情况处理
